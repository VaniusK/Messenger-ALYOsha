## WebSocket
`ws://api.*hostname*/ws/chat` \
`wss://...` Если используется HTTPS
### handleNewConnection - Открытие подключения
При открытии подключения нужно передавать токен в таком же формате, как для защищенных http-запросов. После этого сможем получать новые уведомления о событиях от сервера.
### handleConnectionClosed - Закрытие подключения
Закрывает подключение
### notifyUser - Пересылает сообщение о событии получателю
Сообщения приходят как обычные строки, но оформленные как Json, поэтому их нужно конвертировать на стороне клиента.
  - **Типы событий(поле 'event_type')**:
    - 'NEW_MESSAGE': новое сообщение(Поля сообщения от вебсокета: ['data']['message']: message)
    - 'MESSAGE_READ': сообщения прочитаны(Поля сообщения от вебсокета: soon)
   
## Защищенные методы
Помечены в документации как `(protected)`
- Для доступа к защищенным методам необходимо передавать заголовок:
`Authorization: Bearer <access_token>`

## Общие коды возврата
- `500`: Неизвестная ошибка на сервере(ошибка бд)
- `401`: Невалидный токен(у защищенных методов)
- `400`: Невалидный JSON(отсутствует или не хватает полей)

## HTTPS guide
Если вам интересно захостить сервер самим, то либо напишите dast3rr, либо обратитесь к нейросети.(возможно по запросам зрителей будет полный гайд) На текущий момент в файле nginx.conf находятся настройки для HTTPS соединения, которое при деплое требует создания сертификатов. А чтобы использовать обычный HTTP, нужно немного поменять nginx.conf. Однако для локального запуска сервера достаточно будет создать в директории server директорию certbot/conf/live/localhost и туда создать локальный сертификат через openssl командой "openssl req -x509 -nodes -days 365 -newkey rsa:2048 -keyout ./certbot/conf/live/localhost/privkey.pem -out ./certbot/conf/live/localhost/fullchain.pem -subj "/C=RU/L=Local/O=Dev/CN=localhost" 

## Auth

### POST /auth/register
Регистрация нового пользователя.

- **Аргументы (JSON)**:
  - `handle`: string (уникальный никнейм)
  - `display_name`: string (отображаемое имя)
  - `password`: string (пароль)
- **Коды возврата**:
  - `201`: Успешно
  - `409`: Пользователь с таким handle уже существует

### POST /auth/login
Авторизация пользователя.

- **Аргументы (JSON)**:
  - `handle`: string
  - `password`: string
- **Возвращает**:
  - `token`: string (JWT)
- **Коды возврата**:
  - `200`: Успешно
  - `401`: Неверный пароль или пользователь не найден

## Attachments
Работа с вложениями реализована через S3-хранилище (Yandex Cloud Storage или аналогичное). 
Используется паттерн **"Upload-First"**: файлы сначала загружаются в облако напрямую клиентом, а затем привязываются к сообщению при его отправке.

При загрузке истории чатов, у сообщений есть поле `attachments` — это список уже загруженных и привязанных вложений со всей необходимой информацией (включая ссылки для скачивания).

### Шаг 1. POST /chats/attachments/presigned-links (protected)
Получить ссылки (presigned URLs) для прямой загрузки файлов в S3. Этот запрос нужно делать **до** отправки самого сообщения.

- **Аргументы (JSON)**:
  - `chat_id`: int64
  - `message_type`: string (text, media, voice, etc. См. документацию БД)
  - `files`: list[]:
      - `original_filename`: string
      - `file_size_bytes`: int64
- **Возвращает**:
  - `attachments`: list[]:
       - `upload_url`: string (Ссылка для PUT-запроса в хранилище)
       - `attachment_key`: string (Путь в хранилище)
       - `content_type`: string (MIME-тип файла, определенный сервером)
       - `token`: string (JWT-токен. **Обязательно сохранить для Шага 3!**)
- **Коды возврата**:
  - `200`: Успешно
  - `400`: Несовпадение типов файлов и message_type
  - `403`: Ошибка доступа (нет прав на чат)

### Шаг 2. Загрузка файла в хранилище (Client -> S3)
Выполните HTTP `PUT` запрос по ссылке из поля `upload_url`, полученного на Шаге 1. 
- **Заголовки:** Обязательно укажите `Content-Type`, передав в него значение из поля `content_type` (с Шага 1).
- **Тело запроса:** Сырые бинарные данные файла.
- Дождитесь статуса `200 OK` от облака.

### Шаг 3. Финальная отправка сообщения
После успешной загрузки всех файлов, отправьте сообщение через `POST /chats/{chat_id}/messages`.
- В поле `type` передайте тот же тип, что и в `message_type` на Шаге 1.
- В поле `attachment_tokens` передайте массив строк — `token` (с Шага 1) для каждого успешно загруженного файла. Сервер проверит токены и безопасно привяжет файлы к новому сообщению.

### Загрузка файла из хранилища (Скачивание)
Для скачивания или отображения файла используйте GET-запрос по ссылке из поля `download_url` внутри объекта вложения (массив `attachments` у сообщения, у каждого есть поля `download_url` и `s3_object_key`). 
Ссылка генерируется сервером автоматически и содержит необходимые заголовки для корректного скачивания файла с правильным именем.

## Матрица совместимости (Маппинг интерфейса)

Фронтенд **обязан** проверять форматы файлов на своей стороне перед запросом к API. Тип отправляемого сообщения (`message_type`) зависит от того, что именно и как пользователь хочет отправить.

| Действие пользователя в UI | `message_type` для API | Разрешенные форматы (MIME) | Как рендерить при получении |
| :--- | :--- | :--- | :--- |
| **Голосовое сообщение** | `voice` | Строго `audio/*` | Плеер (кнопка Play/Pause, таймлайн) |
| **Видео-кружочек** | `round` | Строго `video/mp4` | Круглый плеер с автовоспроизведением |
| **Обычные фото/видео** | `media` | Строго `image/*` или `video/*` | Галерея / Большое превью в чате |
| **Отправить любой документ** | `text` | Любые (в т.ч. pdf, docx, zip) | Прямоугольный блок с иконкой файла |
| **Отправить фото "КАК ФАЙЛ"** | `text` | `image/*` (оригинальный тип) | Прямоугольный блок с иконкой файла |
| **Обычный текст без файлов** | `text` | Вложений нет | Стандартный текстовый бабл |
